## Cross-cutting issues (affect multiple labs)

### 1. Duplicated discussion / prompt text (migration artifact)

The auto-migration script doubled every multi-line concept prompt. Affects:
`staticElectricity`, `chargesFields`, `capacitors`, `dcCircuits`, `geometricOptics`, `snellsLaw.draft`.

Pattern in every case: the `prompt` string contains the paragraph twice in a row, separated by a blank line. Each affected field needs the second copy removed.

**Example (`dcCircuits` discussion):**
```ts
prompt: `Write your discussion & conclusion below...

Write your discussion & conclusion below...`
```

### 2. `Symbol(unevaluable)` column units (migration artifact)

Affects `staticElectricity`, `chargesFields`, `dcCircuits`, `snellsLaw.draft`. The migration script emitted `unit: 'Symbol(unevaluable)'` when it could not parse the legacy unit. These need to be replaced with the real unit strings (e.g. `'nC'`, `'N'`, `'V'`, `'A'`) or `unit: 'Symbol(unevaluable)'` should be deleted so the column inherits no unit.

### 3. Heading-inside-prompt anti-pattern

Several concept sections embed a markdown heading (`##` or `### `) inside the `prompt` string. Headings inside prompts are not rendered as headings by the worksheet UI; they appear as literal text starting with `##`. Strip the headings and, if needed, add an `instructions` block before the concept section.

Affected: `chargesFields` (4 prompts), `capacitors` (2 prompts), `dcCircuits` (1 prompt), `geometricOptics` (1 prompt), `snellsLaw.draft` (1 prompt).

### 4. `## Observations:` / `#### Concept Check Questions:` heading levels

Several auto-migrated labs use `##` or `####` headings for sub-headings inside a single Part, inconsistently with the `## Part N` convention. Prefer `## Background:` for theory blocks, `## Part N` for procedural headings, and no heading-level markers on sub-section labels (just bold text).

### 5. Self-referential `imageId`/`captionFieldId` collision

Three image sections use the same string for both `imageId` and `captionFieldId`:
- `geometricOptics` and `convergingLens`: `imageId: 'part1B_screenshot'`, `captionFieldId: 'part1B_screenshot'`
- `geometricOptics` and `divergingLens`: `imageId: 'part2B_screenshot'`, `captionFieldId: 'part2B_screenshot'`

The caption field should have a distinct id (e.g. `captionFieldId: 'part1B_screenshotCaption'`).

### 6. Missing `points` on `measurement` and `multiMeasurement` sections

Many auto-migrated `measurement` and `multiMeasurement` sections have no `points` field. Where the section is graded, the field should be added. Where it is informational (just recording a given value), `points: 0` or omission is fine -- but be consistent.

## Finding 1 (HIGH, kirchhoffsLaws) -- calculus and an unsupported fit in an algebra-based course

`kirchhoffsLaws` is enabled in PHY 114 by direct reuse of the PHY 132 object. The manifest
comment justifies reuse as "no uncertainty content to strip." But uncertainty was never the
divergence for this lab. Two problems:

1. **Calculus content.** Part 2B section `maximumPowerTransferDerivation` (2 pts) instructs:
   "Use calculus to find the theoretical prediction for the condition for maximum power
   transfer... given the function P_R(R), what should dP_R/dR equal at the maximum?" PHY 114 is
   algebra-based (CLAUDE.md). `docs/archive/LEGACY_PARITY_INVENTORY.md` section E confirms the
   legacy PHY 114 dcCircuits **dropped Part 2B entirely** (the power-delivery / max-power-transfer
   half was PHY 132 only). Reusing the 132 object reintroduces to PHY 114 the exact content the
   legacy course excluded: the maximum-power-transfer theorem, a calculus derivation, and the
   Stereo System synthesis problem.

2. **Unsupported fit.** Part 2B procedure step 5 tells students to "Fit your data to the model
   P_R = A R / (R + B)^2" and record A, B. The `powerGraph` plot only offers `linear` and
   `proportional` fits (and `GRAPHING_EXPANSION_SPEC.md` declares custom/nonlinear fits out of
   scope permanently). So the platform cannot perform the fit the lab asks for, and the two fits
   it does offer are both wrong for a peaked curve. This affects **PHY 132 too**, independent of
   the calculus question.

   Suggested resolution: the cleanest split mirrors the legacy intent -- a PHY-114-owned
   `kirchhoffsLaws` containing Part 2A (KVL/KCL, fully algebra-based and platform-supported) that
   drops Part 2B, leaving Part 2B to PHY 132. Confirm with the instructor whether PHY 114 at ASU
   now includes any calculus before deciding; if it deliberately does, only the fit gap (2) needs
   addressing.
